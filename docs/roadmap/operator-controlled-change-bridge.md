# Operator-Controlled IBM i Change Bridge (Roadmap)

Status: proposal and safe scaffolding only (no remote write/apply/compile execution in this branch).

## 1. Purpose

Zeus RPG PromptKit already supports evidence-first workflows:

1. fetch from IBM i
2. normalize locally
3. analyze locally
4. enrich with DB2 metadata
5. generate reports, AI context, and review artifacts

The change bridge extends this path for a future, explicit opt-in scenario:

- move reviewed local source changes toward IBM i targets
- keep control with the operator
- enforce planning, validation, review, approval, and audit before any sensitive step

This is intended to reduce manual coordination risk without changing Zeus into an autonomous deployment tool.

## 2. Non-goals

This bridge is not:

- autonomous deployment
- a VS Code extension feature
- an IBM i IDE
- a replacement for developer/operator review
- a production release manager
- arbitrary CL command execution
- an AI-controlled write system
- a bypass for existing IBM i change management

## 3. Safety Principle

Product-level bridge rule set:

Read by default.  
Change locally.  
Plan before action.  
Diff before upload.  
Stage before replace.  
Compile only through approved templates.  
Audit everything.  
The operator remains responsible.

## 4. Proposed Workflow

1. Fetch source members and/or IFS sources from IBM i.
2. Analyze locally with existing `analyze`, `impact`, `assess-risk`.
3. AI and/or developer proposes local source edits.
4. Run `diff` against last fetched baseline.
5. Run `assess-risk` for the affected program/member.
6. Generate a bridge change plan artifact.
7. Operator reviews the plan and checklist.
8. Optionally stage source to IBM i staging target.
9. Operator explicitly approves apply.
10. Optionally compile by approved template.
11. Collect compile diagnostics.
12. Generate bridge report and append audit trail.
13. AI may suggest fixes, but never self-approve dangerous steps.

## 5. Proposed CLI Shape

Chosen shape: grouped subcommands under `bridge` to avoid top-level command sprawl and keep lifecycle steps explicit.

### `bridge plan`

- Purpose: build local plan artifacts (`change-plan.json`, `change-plan.md`) from local source + validated target mapping.
- Required options: `--profile`, `--program`, `--source`, target options (`--target-lib`, `--target-file`, `--target-member`) or IFS target.
- Safety behavior: validates target type, name format, and allowlist; no remote mutation.
- Refusal behavior: refuses when bridge disabled or target not allowlisted.
- Output artifacts: plan JSON/Markdown plus audit event.
- JSON output: supported via `--json`.
- Relationship: designed to run after `diff`, `assess-risk`, `generate-checklist`.

### `bridge stage`

- Purpose: future upload to staging target only.
- Required options: `--profile`, `--program`.
- Safety behavior: dry-run supported; no production overwrite path.
- Refusal behavior: refuses when bridge disabled; refuses non-dry-run in current scaffold.
- Output artifacts: audit event (and in future `stage-report.json` / `.md`).
- JSON output: supported via `--json`.
- Relationship: consumes plan artifacts generated by `bridge plan`.

### `bridge apply`

- Purpose: future controlled replace/apply after staging.
- Required options: `--profile`, `--program`.
- Safety behavior: requires explicit approval and plan checks by design.
- Refusal behavior: refuses when bridge disabled; refuses non-dry-run in scaffold.
- Output artifacts: audit event (future apply report).
- JSON output: supported via `--json`.
- Relationship: separate command to preserve review gate between stage and apply.

### `bridge compile-plan`

- Purpose: validate compile template selection and compile metadata.
- Required options: `--profile`, `--program`, `--template <id>`.
- Safety behavior: template ID must be known and allowlisted; arbitrary command text rejected.
- Refusal behavior: refuses when compile bridge disabled or template not valid.
- Output artifacts: audit event (future compile-plan artifact).
- JSON output: supported via `--json`.
- Relationship: explicit step before any compile-run action.

### `bridge compile-run`

- Purpose: future template-driven compile execution after approval.
- Required options: `--profile`, `--program`, `--template <id>`.
- Safety behavior: `--dry-run` default path; command text injection blocked.
- Refusal behavior: refuses non-dry-run in scaffold.
- Output artifacts: audit event (future compile report).
- JSON output: supported via `--json`.
- Relationship: follows `bridge compile-plan` and operator approval.

### `bridge report`

- Purpose: show expected bridge artifact locations for a program.
- Required options: `--profile`, `--program`.
- Safety behavior: read-only local output.
- Refusal behavior: same bridge-enabled gate.
- Output artifacts: none (read-only pointer output).
- JSON output: supported via `--json`.
- Relationship: convenient artifact discovery for reviews and tickets.

## 6. Configuration Model

Bridge configuration is profile-based and disabled by default.

Reference model:

```json
{
  "bridge": {
    "enabled": false,
    "mode": "plan-only",
    "requireConfirmation": true,
    "allowAutoApprove": false,
    "auditLog": true,
    "allowedTargets": {
      "libraries": [],
      "sourceFiles": [],
      "ifsPaths": []
    },
    "staging": {
      "enabled": true,
      "library": "",
      "sourceFile": "",
      "ifsPath": ""
    },
    "compile": {
      "enabled": false,
      "allowedTemplates": [],
      "requirePlan": true,
      "requireApproval": true
    }
  }
}
```

Behavior notes:

- `bridge.enabled=false` blocks all bridge commands.
- `compile.enabled=false` blocks compile-plan/run commands.
- allowlists are mandatory for target admission.
- values are validated during profile loading and runtime normalization.

## 7. Target Model

Supported conceptual target types:

1. `source-member`
2. `ifs-streamfile`

For `source-member`:

- library: validated IBM i object name
- source physical file: validated IBM i object name
- member: validated IBM i object name
- member type: optional validated object-style token

For `ifs-streamfile`:

- absolute IFS path only
- no parent traversal (`..`)

Normalization and safety:

- object names normalized to uppercase
- invalid characters rejected
- allowlist enforced (libraries/sourceFiles/ifsPaths)
- staging target must be validated independently from production target
- future model includes remote version token (hash or timestamp) for conflict detection
- future model includes CCSID and line-ending normalization metadata before upload

## 8. Write-back Planning

Planned artifact location:

- `output/<PROGRAM>/change-plan.json`
- `output/<PROGRAM>/change-plan.md`
- `output/<PROGRAM>/bridge-approval.json`
- `output/<PROGRAM>/bridge-approval.md`

Current scaffold already writes these locally.

Plan fields:

- planId
- planHash
- local source path
- remote target + target type
- before/after hashes (if provided)
- diff summary
- risk level
- required approvals
- intended action
- staging destination summary
- rollback hints
- generated timestamp
- profile name (without secrets)
- warnings

## 9. Staging Model

Staging is the first remote mutation step by design, and should:

- never overwrite production source directly
- target staging library/source file or staging IFS path
- compute and record content hash before transfer
- verify post-transfer content hash where possible
- emit staging report artifacts
- require explicit apply approval after staging

Current branch status:

- no remote staging execution implemented
- dry-run + audit scaffolding only

## 10. Apply Model

Apply is explicitly separate from staging and must:

- refuse when bridge disabled
- refuse without plan
- refuse non-allowlisted target
- refuse on remote drift unless override policy is explicit and auditable
- refuse when approval state missing
- log every action
- include backup/rollback hints
- avoid member delete behavior in MVP

Current branch status:

- apply execution intentionally not implemented
- refusal + dry-run audit scaffold only

## 11. Compile Model

Compile support is template-driven only.

Guardrails:

- no arbitrary CL command text accepted from AI/CLI free text
- compile template IDs must be from known set
- template ID must be allowlisted by profile when configured
- compile bridge can be disabled independently

Template IDs scaffolded:

- `crtbndrpg`
- `crtrpgmod`
- `crtclpgm`
- `crtsqlrpgi`

Future compile validation contract:

- validate libraries, object names, source files, members, options
- enforce output-object naming rules
- include target release/profile constraints where needed

Flow:

1. `bridge compile-plan`
2. operator review
3. `bridge compile-run` (approved)
4. collect diagnostics
5. write `compile-report.json` + `compile-report.md`

Current branch status: validation + refusal scaffolding only.

## 12. Diagnostics

Target diagnostic structure for compile/apply jobs:

- severity
- messageId
- messageText
- source member
- local source path
- line number and column when mappable
- compileTemplateId
- job/user/timestamp where available
- suggested next action

In this branch, only audit events are emitted; compile/joblog parsing is deferred.

## 13. Agent Contract

Bridge usage contract for AI agents:

- agent may analyze artifacts
- agent may propose local edits
- agent may request plan generation
- agent may summarize diffs
- agent may explain risk
- agent may suggest compile fixes
- agent must not self-approve stage/apply/compile actions
- operator remains responsible for approvals and execution

Even in assisted mode, approval ownership stays human.

## 14. Audit Log

Proposed audit stream:

- `output/audit/bridge-audit.jsonl`

Event fields:

- timestamp
- command
- profile
- actor mode (`human`, `agent-assisted`, `automation`)
- local source
- remote target
- action
- dry-run flag
- approval status
- result
- warnings
- masked configuration summary
- content hashes where available

Current scaffold:

- writes JSONL events locally
- masks secrets using existing `src/security/secretMasking.js`

## 18. Review Gate Before Real Remote Execution

Before implementing actual IBM i write/apply/compile operations:

1. human review of this roadmap and scaffold
2. explicit approval of command contract and approval model
3. explicit approval of audit schema and rollback contract
4. explicit approval of template catalog and validator strictness

No remote mutation execution should be merged before this review gate is passed.
